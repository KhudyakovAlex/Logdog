# LLM

## Постановка задачи (Logdog)

Нужен постоянно работающий локальный сервер в домашней/офисной LAN, который **собирает отладочные логи с любых приложений**.
Приложения должны уметь отправлять лог-записи по HTTP. Сервер должен сохранять записи и давать доступ к ним для Cursor AI
через **MCP**, чтобы агент мог подтягивать свежие логи при отладке.

### Контекст и окружение

- **ОС**: Windows 10
- **Запуск**: Docker
- **Сеть**: локалка (доверенная), **без аутентификации**
- **Порты**: HTTP ingest `3000`

### Интеграция с Cursor

- Cursor должен получать логи через **MCP по stdio** (принятый вариант): Cursor запускает локальный MCP-процесс.
- MCP читает ту же SQLite БД, что и ingest-сервер.

### Хранение

- **Хранилище**: SQLite (один файл)
- **Размещение БД**: на хосте, в примонтированной папке, например `./data/logdog.db`
- **Режим**: WAL для конкурентного чтения (MCP) и записи (HTTP ingest)

### Ротация/retention

- Ограничение размера БД: **1 GB**
- Поведение при превышении лимита:
  - удалять самые старые записи до возврата под лимит
  - выполнять операции, позволяющие реально уменьшать файл БД (например, incremental vacuum)

### Формат лог-события (JSON)

Сервер принимает лог-запись как JSON (одна запись за запрос). Поля:

- `ts`: timestamp (если не передан — проставляется сервером)
- `level`: уровень (`debug|info|warn|error`)
- `app`: имя приложения/сервиса
- `message`: строка
- `traceId` (опционально): идентификатор трассы/корреляции
- `fields` (опционально): произвольные доп. поля (JSON object)

Ограничение размера одной записи: **до 256KB** (включая stacktrace/fields).

### HTTP ingest API (минимум)

- `POST /logs`
  - body: JSON события (см. выше)
  - response: подтверждение приёма + id записи (или аналог)

### MCP API (tools)

MCP должен предоставлять инструменты для агентов:

- `recent(limit, app?, level?)`
  - вернуть последние N записей (с возможностью фильтра по `app`, `level`)
- `query(app?, level?, since?, until?, contains?, traceId?, limit)`
  - поиск по фильтрам:
    - `app`, `level`
    - диапазон времени `since..until`
    - `contains`: подстрока в `message` (и/или сериализованных fields — уточняется реализацией)
    - `traceId`
    - ограничение количества `limit`

### Нефункциональные требования

- Сервер должен быть “всегда включен” (контейнер) и переживать рестарт.
- Запись логов должна быть устойчивой: при ошибках входные данные валидируются, сервер не падает.
- MCP-запросы не должны блокировать ingest (и наоборот).

### Не входит (на текущем этапе)

- Аутентификация/авторизация
- Внешние системы логирования (Elastic/Loki/Seq)
- UI/веб-интерфейс просмотра логов
